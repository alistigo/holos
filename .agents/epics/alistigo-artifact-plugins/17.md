# Task: Refactor @alistigo/artifact-list to load plugins

## Description

Remove the hardcoded Sentry/PostHog wiring from `alistigo-artifact-list` and replace it
with the generic plugin-loading mechanism: read `options.plugins` from `MountOptions`,
dynamically import each named plugin, run its lifecycle hooks at the right points in the
existing bootstrap sequence, and replace the `@sentry/react` `ErrorBoundary` with a
host-owned generic one that emits an `error:uncaught` event.

This is the highest-risk task in the epic — it changes `mount()`'s signature (sync →
async) and removes two dependencies (`@sentry/react`, and effectively `@sentry/browser`/
`posthog-js` too). Full integration testing (a real Sentry/PostHog plugin actually
loading) needs tasks 003 and 004 to exist, though this task's own unit tests can use a
fake in-memory `AlistigoPlugin`.

## Acceptance Criteria

- [ ] Delete `src/monitoring.ts` and `src/analytics.ts` (and their tests)
- [ ] New `src/plugins.ts`: `loadPlugins(spec: Record<string, Record<string, unknown>> | undefined)` → `Promise.allSettled` over `Object.entries(spec ?? {})`, calling `loadPlugin(pkg)` from `@alistigo/artifact-plugin-api`; failures are logged (not thrown) and dropped from the returned plugin list
- [ ] `MountOptions` (in `mount.ts`) gains `plugins?: Record<string, Record<string, unknown>>`
- [ ] `mount()` becomes `async function mount(container, options = {}): Promise<void>`; sequence: resolve container → `registerMount()` → load plugins → `createPluginRuntime(...)` → `runtime.setup()` → `bootI18n()` → `runtime.beforeMount()` → `renderApp()` → `runtime.mounted()` → if first mount, `runtime.bus.emit("widget:displayed", {...})`
- [ ] New small host-owned `ErrorBoundary` React component (plain `componentDidCatch`, no `@sentry/react` import) wraps `<App>` in `renderApp()`; on catch, emits `error:uncaught` with `{ error, componentStack }` on the plugin runtime's bus, and renders the existing `<Fragment/>` fallback
- [ ] `index.tsx`: remove the `initMonitoring(pkg.version)` call that runs before `autoMount()` — plugin `setup()` now happens inside `mount()` instead; update the module's JSDoc to note the narrower error-capture window is an accepted tradeoff (Sentry only initializes once a plugin is configured and `mount()` runs, not at module-load time)
- [ ] `index.tsx`'s documented "explicit mount (advanced)" JSDoc updated to reflect `mount()` is now async
- [ ] Remove `@sentry/browser`, `@sentry/react`, `posthog-js` from this package's `package.json` dependencies; remove `@sentry/vite-plugin` and its config from `vite.config.ts` (moved to the sentry plugin package in task 003)
- [ ] Existing tests updated/passing: `auto-mount.ts`'s call into `mount()` now needs to be awaited where relevant
- [ ] `pnpm build && pnpm build:typecheck` pass
- [ ] `pnpm nx qa:lint` passes

## Technical Details

Read `packages/alistigo-artifact-list/src/mount.ts`, `index.tsx`, `monitoring.ts`,
`analytics.ts`, and `auto-mount.ts` fully before starting — the exact current bootstrap
order (registerMount → bootI18n → initAnalytics → renderApp → trackWidgetDisplayed) is
what the new `runtime.*()` calls must slot into, not a redesign of the ordering itself.

Every plugin lifecycle call inside `createPluginRuntime` (from task 001) already
try/catches internally — `mount()` itself does not need its own additional error
handling around `runtime.setup()` etc.

# Task: Add plugins field to @alistigo/artifact-config-format

## Description

Add a `plugins` field to `@alistigo/artifact-config-format`'s types, validator, and JSON
Schema, so a config document can declare `plugins: { "<pkg-name>": {...} }`. Validation
stays shallow — it must not import or validate against any individual plugin's own schema.

## Acceptance Criteria

- [ ] `src/types.ts`: `ArtifactConfig` gains `plugins?: Record<string, Record<string, unknown>>`
- [ ] `src/validate.ts`: new `assertPluginsField(raw)` — throws `TypeError` if `plugins` is present but not a plain object, or if any value under it is not a plain object; does not inspect deeper
- [ ] `schemas/artifact-config.json`: `plugins: { "type": "object", "additionalProperties": { "type": "object" } }` added to the **top-level `properties`**
- [ ] Same `plugins` schema fragment also added to the `then.properties` block for `@alistigo/artifact-list` (that branch has `"additionalProperties": false`, so omitting this causes any list config with `plugins` set to fail validation)
- [ ] `plugins` is NOT added to `KNOWN_ARTIFACTS` or `dispatchArtifactValidator` — it is orthogonal to the artifact-type discriminated union
- [ ] Unit tests: valid `plugins` object passes; non-object `plugins` value throws; a `@alistigo/artifact-list` config with `plugins` set passes both `then.additionalProperties: false` and the new field
- [ ] `pnpm build` and `pnpm build:typecheck` pass
- [ ] `pnpm nx qa:lint` passes

## Technical Details

Read `packages/alistigo-artifact-config-format/src/types.ts`, `src/validate.ts`, and
`src/schemas/artifact-config.json` before editing — the `then`/`else` `if` branch keyed
on `app === "@alistigo/artifact-list"` is the one place the double-edit is easy to miss.
